# 项目整体分析

## Summary

基于当前仓库勘察，这个项目是一个 **Next.js App Router + React + TypeScript + Tailwind CSS v4** 的前端原型，主题是“校园信息检索与可解释问答助手”。

它当前 **没有真实后端、没有 API Route、没有数据库或向量检索链路**，核心是用 `mockSearchProvider` 模拟 RAG 产品体验，重点放在前端交互、来源分层、结果可解释性和信息展示。

从项目形态看，它已经具备完整主流程，适合作品集展示；从工程成熟度看，它仍然是演示级原型，还不具备真实 RAG 产品的可用性和可信度。

## 项目定位与目标用户

这个项目不是“聊天助手”，而是“检索优先、回答可追溯”的校园信息查询产品原型。

它的核心卖点不是模型生成能力本身，而是下面几件更接近真实信息产品的事情：

- 回答和证据一起展示
- 官方信息和社区经验明确分层
- 命中片段可以直接核验
- 信息不足时不强行生成答案

更适合的使用场景和受众包括：

- 想快速查询校园规则、办事流程和生活信息的学生
- 想做一个有 AI 产品感、但仍突出前端价值的作品集项目的开发者
- 需要在面试中讲“可解释回答”和“复杂状态设计”的前端候选人

## 页面流与交互主流程

### 首页 `/`

首页负责建立产品认知，并把用户引导到主流程。

- `HomePage` 负责产品定位、亮点和数据策略的呈现
- `SearchBox` 负责输入问题和跳转搜索页
- `SuggestedQuestions` 提供默认问题，降低首屏使用门槛
- `HistoryPanel` 展示本地搜索历史，支持重查和清空

这个页面本质上承担两件事：

- 让用户理解“这不是聊天，而是检索产品”
- 给用户一个低摩擦的第一步

### 结果页 `/search?q=...`

结果页承载整个搜索和回答体验。

- URL 参数 `q` 作为问题输入
- `ResultsShell` 负责整页状态编排
- 页面先经历 `retrieving` 再经历 `summarizing`
- 完成后展示回答、来源、筛选和相关问题

页面上的主要交互包括：

- `StatusPanel` 显示当前检索阶段
- `ResultToolbar` 切换来源筛选与视图模式
- `AnswerPanel` 展示结构化回答
- `SourceList` / `SourceCard` 展示来源卡片和命中片段
- `RelatedQuestionsPanel` 提供进一步追问入口和结果说明
- `EmptyState` 在无结果时承接兜底

## 状态流

这个项目的状态复杂度不高，但状态组织得比较完整，适合当作前端案例来讲。

- 输入状态：由 `SearchBox` 本地 state 管理
- 导航状态：由 `useSearchNavigation` 负责 URL 跳转和同 query 刷新
- 历史状态：由 `SearchHistoryProvider` 写入 `localStorage`
- 搜索结果状态：由 `ResultsShell` 管理 `response`
- 视图状态：由 `ResultsShell` 管理 `filter`、`viewMode` 和 `expandedSourceIds`
- 加载阶段：`idle -> retrieving -> summarizing -> done`

这个状态拆法的优点是：

- 搜索行为和 UI 展示分开
- 历史记录可以在首页和结果页复用
- 将来替换真实接口时，只需要换数据提供层，不必重构页面壳

## 架构分层

### 1. 视图层

视图层主要负责把产品意图表达清楚，不承担复杂数据策略。

- `HomePage`：首页信息组织
- `ResultsShell`：结果页主壳和状态编排
- `SearchBox`：搜索输入和提交
- `StatusPanel`：展示加载和结果状态
- `ResultToolbar`：来源过滤和视图切换
- `AnswerPanel`：结构化回答展示
- `SourceList` / `SourceCard`：来源列表、来源展开和关键词高亮
- `RelatedQuestionsPanel`：相关问题和可信度提示
- `EmptyState`：无答案兜底

### 2. 导航与状态层

这一层把页面行为串起来。

- `useSearchNavigation`：处理搜索提交、URL 跳转和同 query 刷新
- `SearchHistoryProvider`：维护本地搜索历史
- `ResultsShell`：连接搜索、loading、筛选和视图状态

### 3. 数据适配层

这是当前项目最像真实 RAG 系统接口边界的部分。

- `src/lib/search/types.ts` 定义统一搜索数据契约
- `SearchProvider.search(query)` 抽象统一搜索接口
- `mockSearchProvider` 用关键词匹配模拟召回和回答生成
- `mock-data.ts` 提供场景、来源、相关问题和回答内容

这个分层有一个明显优点：未来接入真实接口时，只要后端返回结构满足 `SearchResponse`，大部分页面代码都可以复用。

## 当前亮点

### 问答 / 检索双模式

这个项目没有把 AI 回答当作唯一输出，而是把“回答”和“原始检索片段”并列展示。用户既可以快速看摘要，也可以切到检索结果视图直接审查证据。

### 官方 / 社区来源分层

事实类信息优先看官方来源，经验类补充看社区内容。这个设计可以有效降低“经验被包装成权威事实”的误导风险。

### 引用来源可追溯

回答不是悬空文本。每条回答都能在来源卡片中找到依据，来源卡片同时提供来源类型、发布时间、片段和命中关键词。

### 无答案兜底

系统在信息不足时不会硬生成一个看起来完整的答案，而是返回空结果或部分命中，并引导用户继续细化提问。

## 工程判断

### 为什么它适合作品集展示

- 主题鲜明，场景具体
- 页面数量少，但主流程闭环完整
- 包含 loading、空态、筛选、双视图、来源展开和历史记录等可讲状态
- UI 明确带有产品表达，不是单纯技术 demo
- AI 元素存在，但没有盖过前端设计和交互价值

### 为什么它仍然只是演示级原型

- 数据全部来自静态 mock
- 搜索逻辑只是简单关键词命中
- 置信度和 `generatedAt` 由前端拼装，不代表真实质量
- 没有真实错误态、埋点、监控和测试体系
- 没有用户体系，也没有跨端同步能力

## 风险与短板

### 1. 依赖版本全部使用 `latest`

`package.json` 当前的依赖版本策略会降低环境可复现性，也会让后续安装和构建结果随时间漂移。

### 2. 搜索能力只是 mock 占位

当前搜索逻辑不具备真实 RAG 系统需要的能力：

- 多源召回
- 结果去重
- 时效控制
- 可信度建模
- 片段排序和重排序

### 3. `generatedAt` 的语义存在误导风险

当前字段更接近“页面生成时间”，而不是“来源更新时间”或“检索数据时间”。接入真实后端后，这个字段需要重新定义。

### 4. 缺少真实服务边界

现在只有 `ok`、`partial` 和 `empty` 三种结果状态，没有覆盖网络错误、接口超时、服务降级、脏数据等真实场景。

### 5. 缺少测试和可观测性

虽然项目已经能完成完整交互，但没有自动化测试、埋点、监控和日志，工程上还不够稳。

## 后续演进方向

### 第一阶段：替换真实数据层

- 保留现有 `SearchProvider` 契约
- 用真实 API 替换 `mockSearchProvider`
- 让回答、来源、相关问题和结果状态由后端统一返回

### 第二阶段：补产品可靠性

- 增加错误态和重试策略
- 区分“数据时间”和“生成时间”
- 为来源卡片补充外链跳转和更多可信度说明
- 加入埋点，观察搜索和筛选行为

### 第三阶段：补工程基础设施

- 固定依赖版本
- 增加组件测试和主流程测试
- 增加接口契约和数据校验
- 为 mock 数据和真实数据建立更清晰的替换边界

## 数据策略

v1 版本只使用小规模、高质量 mock 数据来完整演示前端体验，不直接接入真实爬虫或生产级 RAG 服务。

当前数据分为两类：

- 官方数据：图书馆规则、住宿 FAQ、社团活动说明、食堂营业时间等
- 社区数据：经验帖、常见问题整理、排队与使用建议

当前数据清洗策略：

- 去重
- 去广告
- 去联系方式
- 保留时间信息
- 标记来源类型

## Validation

基于当前仓库检查，可以确认下面这些事实：

- 项目结构是单体前端仓库，不是前后端混合 monorepo
- 核心代码位于 `src/app`、`src/components`、`src/lib/search`
- 当前没有真实 API Route 或服务端数据接入点
- 搜索结果来自 `src/lib/search/mock-data.ts` 和 `src/lib/search/search-provider.ts`
- 生产构建已经通过 `npm run build`

- 分析输出更偏架构、产品和工程质量，而不是面试背稿
