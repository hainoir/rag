# 项目面试讲稿

## 1 分钟介绍

这个项目是一个校园信息检索与可解释问答助手，我把它做成了一个 RAG 产品原型，而不是普通聊天页面。  
核心思路是让用户在提问后，不只是看到一段 AI 回答，而是同时看到回答摘要、来源分层和原始命中片段，这样信息更容易核验，也更接近真实的信息检索产品。

技术上它是一个基于 **Next.js App Router + React + TypeScript + Tailwind CSS v4** 的前端项目。当前数据层还是 mock，但接口层已经抽象成了 `SearchProvider`，所以后续可以平滑替换成真实后端。

## 3 分钟版本

### 我为什么做这个项目

我不想再做一个“输入问题 -> 输出一大段 AI 文本”的聊天壳子，因为那类项目很难体现前端在产品可信度和信息组织上的价值。

所以我选了一个更像真实产品的问题场景：校园信息查询。它天然会涉及：

- 官方规则和社区经验混在一起
- 用户既想快拿答案，也想核对依据
- 信息不完整时，产品不能乱答

这正好适合用前端交互去体现“可解释回答”。

### 这个项目解决什么问题

它主要解决三个问题：

- 用户怎么快速拿到结论
- 用户怎么确认结论有没有依据
- 系统在信息不够时怎么安全兜底

所以我把结果页设计成两层输出：

- 第一层是结构化回答，快速给结论
- 第二层是来源卡片和原始片段，方便用户验证

### 这个项目最想体现的能力

- 把一个 AI 主题做成真正的产品流，而不是技术演示
- 管理多种前端状态，包括 loading、空态、筛选、双视图和来源展开
- 用清晰的组件拆分和接口抽象，为后续真实化改造留边界

## 可以重点讲的设计点

### 1. 问答 / 检索双模式

我没有把回答当成最终输出，而是允许用户在“回答视图”和“检索结果视图”之间切换。  
这样做的价值是：

- 普通用户先看摘要，效率高
- 谨慎用户可以直接审查原始片段
- 产品的可信度表达更完整

### 2. 官方 / 社区来源分层

来源被明确分成 `official` 和 `community` 两类。  
这样事实类问题优先看官方信息，社区内容只做经验补充，避免把讨论内容包装成确定事实。

### 3. 安全兜底

当前结果状态不是只分“有答案 / 没答案”，而是分成：

- `ok`
- `partial`
- `empty`

这意味着系统在信息不足时不会强行输出一个过度确定的回答，这一点我觉得比“能不能答出来”更接近真实产品要求。

### 4. 数据层提前抽象

虽然现在还没有真实后端，但我没有直接把 mock 数据写死在页面里，而是抽成了 `SearchProvider.search(query)`。  
这样后续接真实 API 时，页面结构和大部分展示逻辑可以保持不动。

## 架构讲法

我会把项目拆成三层来讲：

### 视图层

负责产品体验表达，比如：

- `HomePage`
- `ResultsShell`
- `AnswerPanel`
- `SourceList`
- `SourceCard`

### 导航与状态层

负责把行为串起来，比如：

- `useSearchNavigation` 负责搜索跳转和同 query 刷新
- `SearchHistoryProvider` 负责本地历史记录
- `ResultsShell` 负责整页状态编排

### 数据适配层

负责统一数据结构，比如：

- `SearchProvider`
- `SearchResponse`
- `mockSearchProvider`

这个拆法的好处是，页面逻辑、交互状态和数据来源没有搅在一起，可维护性会更好。

## 如果面试官追问“难点是什么”

可以从这几个角度回答：

### 难点 1：不是功能多，而是状态要完整

项目页面不多，但状态组合很多：

- 搜索输入
- 搜索跳转
- 两阶段 loading
- 结果状态
- 来源筛选
- 视图切换
- 来源展开
- 本地历史

难点在于把这些状态拆清楚，而不是堆在一个组件里。

### 难点 2：AI 产品里前端怎么体现价值

如果只是接一个模型接口然后渲染文本，前端价值很弱。  
所以我更关注的是：

- 信息怎么分层
- 来源怎么展示
- 结果怎么让用户愿意信
- 不确定时怎么兜底

### 难点 3：原型和真实产品之间要留迁移边界

当前是 mock 数据，但我不希望以后接真实 API 时推翻重来，所以一开始就把搜索接口和返回结构抽象出来了。

## 如果面试官追问“还不够好的地方”

这个项目目前最大的不足有几个：

- 依赖版本使用 `latest`，可复现性不够稳
- 搜索逻辑只是关键词命中，不是真实 RAG
- `generatedAt` 更像页面生成时间，不是真实数据时间
- 没有错误态、埋点、监控和自动化测试
- 没有用户体系，也没有跨端同步

如果继续做，我会优先补三件事：

1. 接入真实 API，保留 `SearchProvider` 契约不变
2. 补错误态、外链来源、结果时间语义和埋点
3. 固定依赖版本并补测试

## 一句话总结

如果只用一句话总结这个项目，我会说：

> 这是一个把 RAG 主题做成前端产品原型的项目，我重点解决的不是“模型会不会回答”，而是“回答如何更可信、更可核验、更像真实产品”。

## 建议你现场怎么讲

最稳的讲法是按这个顺序：

1. 先讲为什么不做聊天壳子，而做检索产品
2. 再讲首页和结果页的主流程
3. 然后讲问答 / 检索双模式、来源分层和安全兜底
4. 最后讲 `SearchProvider` 抽象和后续如何接真实后端

这样讲的好处是，产品思路、前端价值和工程边界都会比较清楚。
