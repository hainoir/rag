# 项目面试讲稿

这份讲稿默认面向 **前端 / Web 岗位**，目标不是把自己包装成做过生产级 RAG 平台的人，而是把这个项目讲成一个 **可解释 RAG 的前端产品原型**。

## 1 分钟介绍

这个项目是一个校园信息检索与可解释问答助手，我把它做成了一个 RAG 思路驱动的前端产品原型，而不是普通聊天页面。

我想解决的不是“模型能不能说一段话”，而是“用户拿到答案后能不能快速核验”。所以用户提问后，页面不会只展示一段 AI 文本，而是同时展示回答摘要、来源分层和原始命中片段。这样它更像真实的信息检索产品，也更能体现前端在可信度表达上的价值。

技术上它基于 **Next.js App Router + React + TypeScript + Tailwind CSS v4**。前端现在通过 `/api/search` 获取统一的 `SearchResponse`，接口背后由 `searchServiceProvider` 代理到外部搜索服务。这个仓库已经把来源字段、时间语义和状态边界补齐了，但抓取、清洗、去重和索引仍然属于上游服务，不会把它说成完整生产级 RAG 平台。

## 3 分钟版本

### 我为什么做这个项目

我不想再做一个“输入问题 -> 输出一大段 AI 文本”的聊天壳子，因为那类项目很难体现前端在产品可信度和信息组织上的价值。

校园信息查询更像真实产品场景，因为它天然会遇到三类问题：

- 官方规则和社区经验混在一起
- 用户既想快拿答案，也想核对依据
- 信息不完整时，产品不能乱答

所以这个题目很适合用前端交互去体现“可解释回答”。

### 这个项目解决什么问题

它主要解决三个问题：

- 用户怎么快速拿到结论
- 用户怎么确认结论有没有依据
- 系统在信息不够时怎么安全兜底

所以我把结果页设计成两层输出：

- 第一层是结构化回答，先给结论
- 第二层是来源卡片和原始片段，方便用户验证

### 这个项目最想体现的能力

- 把一个 AI 主题做成真正的产品流，而不是技术演示
- 管理多种前端状态，包括 loading、空态、筛选、双视图和来源展开
- 给后续真实化改造预留工程边界，而不是把搜索结果结构直接写死在页面里

## RAG 基础讲法

如果面试官先问“你怎么理解 RAG”，可以按这四步讲：

1. 用户先提问，系统先去检索相关来源，而不是直接让模型裸答。
2. 检索层返回候选片段，最好能区分来源类型和时效性。
3. 回答层基于这些来源组织摘要，并明确告诉用户依据来自哪里。
4. 产品层要把回答和证据一起展示出来，信息不足时不能强答。

接着补一句：

> 我这个项目重点做的是第 3 和第 4 层，也就是回答组织、来源分层和可核验交互；第 1 和第 2 层已经有清晰契约，但抓取和索引服务不在这个仓库里，所以不会把它说成完整生产系统。

## 项目映射讲法

讲“当前项目里哪些是前端表达、哪些属于搜索服务”时，可以直接这样对应：

- 用户提问：`SearchBox` 或推荐问题触发搜索
- 搜索导航：`useSearchNavigation` 负责跳转到 `/search?q=...`
- 统一入口：结果页由 `ResultsShell` 请求 `/api/search`
- 服务端适配：`/api/search` 内部调用 `searchServiceProvider`
- 统一结果：接口返回 `SearchResponse`
- 前端表达：`AnswerPanel` 展示摘要，`SourceList` 展示来源和命中片段
- 可信度控制：通过 `official / community` 分层和 `ok / partial / empty / error` 结果状态做兜底

一句话总结这段映射：

> 当前项目已经有统一 API 边界和来源契约；真正完成度更高的部分在前端信息组织和状态设计，而抓取与索引服务仍然独立于这个仓库。

## 可以重点讲的设计点

### 1. 问答 / 检索双模式

我没有把回答当成最终输出，而是允许用户在“回答视图”和“检索结果视图”之间切换。

这样做的价值是：

- 普通用户先看摘要，效率高
- 谨慎用户可以直接审查原始片段
- 产品的可信度表达更完整

### 2. 官方 / 社区来源分层

来源被明确分成 `official` 和 `community` 两类。

这样事实类问题优先看官方信息，社区内容只做经验补充，避免把讨论内容包装成确定事实。

### 3. 安全兜底

当前结果状态不是只分“有答案 / 没答案”，而是分成：

- `ok`
- `partial`
- `empty`
- `error`

这意味着系统在信息不足时不会强行输出一个过度确定的回答，这一点比“能不能答出来”更接近真实产品要求。

### 4. API 边界先行

前端现在不直接依赖具体搜索实现，而是统一请求 `/api/search`。这个接口背后由 `searchServiceProvider` 代理到外部服务，工程上已经形成了“页面层”和“数据提供层”的边界，后续替换抓取器、索引方式或排序策略时，前端成本会更低。

## 高频追问准备

### 为什么不做聊天壳子

因为校园信息查询不是一个只要“能生成文本”就够的场景。用户更关心答案对不对、依据是什么、能不能自己复核，所以我更想做检索优先、回答可追溯的产品。

### 为什么一定要分 `official` / `community`

因为这两类信息的可信度和使用方式不同。事实类问题应该优先给官方来源，社区内容更适合补充体验和细节。如果不分层，用户会把经验帖误读成确定事实。

### 为什么有 `ok / partial / empty / error`

因为真实信息产品不能把“没找到”和“找到一点点”混成一种状态。`partial` 可以明确告诉用户：有相关线索，但不足以形成高置信回答；`empty` 则表示当前没有足够可靠的信息。

### `resultGeneratedAt` 为什么还要单独强调语义

因为它表示的是这次响应对象生成的时间，而不是来源的更新时间。真实化后应该继续把“结果生成时间”和“来源发布时间 / 数据更新时间”分开。

### 当前项目算不算真实 RAG

不算生产级 RAG。更准确的说法是：它是一个 **RAG 思路驱动的前端产品原型**。现在已经有统一 API 边界、统一结果契约和来源时间元信息，但抓取、清洗、去重和索引服务仍然需要在上游独立实现。

### 如果继续做，你下一步会补什么

我会按这个顺序做：

1. 把来源注册表里的模板域名换成目标校园的真实来源
2. 在独立搜索服务里补抓取、清洗、去重、分块和索引
3. 固定依赖版本并补测试
4. 再补更细的错误分类、埋点和监控

### 如果检索不到、来源冲突、信息过期怎么办

- 检索不到：返回 `empty`，不给过度确定的答案
- 来源冲突：优先展示官方来源，并把社区内容降级为补充说明
- 信息过期：明确标时间，并提醒用户回到原始通知核验

## 演示脚本

建议固定演示 3 个 query，分别覆盖 `ok / partial / empty` 三种主流程状态。

如果你想顺手补一个失败态演示，也可以手动说明：

> 现在项目已经能区分“没有答案”和“检索失败”，不会再把请求失败伪装成空结果。

### 1. `图书馆怎么借书？`

讲法：

- 先看回答摘要，说明它给的是一个可执行结论
- 再切到来源区，强调官方规则和新生指南提供硬依据
- 最后补一句社区内容只是在补充高峰期和续借失败经验

你要讲出的核心是：

> 这个 query 体现的是“先给结论，再给证据”，以及官方 / 社区分层的价值。

### 2. `宿舍洗衣怎么收费？`

讲法：

- 这个问题会更接近 `partial`
- 系统能找到宿舍和洗衣相关线索，但没有足够高置信的直接答案
- 所以前端不会强答，而是提示继续细化问题并展示可参考来源

你要讲出的核心是：

> 这个 query 体现的是安全兜底，不是所有相关命中都应该被包装成确定答案。

### 3. `校园里哪里能打印 3D 模型？`

讲法：

- 这个问题超出了当前来源注册表和搜索服务的覆盖范围
- 系统返回 `empty`，不会伪造来源或编造结论
- 这正好说明产品把“不知道”当作正式状态来设计

你要讲出的核心是：

> 这个 query 体现的是产品可信度来自“知道什么时候不该回答”。

## 如果面试官追问“难点是什么”

可以从这三个角度回答：

### 难点 1：不是功能多，而是状态要完整

项目页面不多，但状态组合很多：

- 搜索输入
- 搜索跳转
- 两阶段 loading
- 结果状态
- 来源筛选
- 视图切换
- 来源展开
- 本地历史

难点在于把这些状态拆清楚，而不是堆在一个组件里。

### 难点 2：AI 产品里前端怎么体现价值

如果只是接一个模型接口然后渲染文本，前端价值很弱。所以我更关注的是：

- 信息怎么分层
- 来源怎么展示
- 结果怎么让用户愿意信
- 不确定时怎么兜底

### 难点 3：原型和真实产品之间要留迁移边界

当前仓库已经有最小官方来源摄取、Postgres 入库、真实验证过的 `lexical / hybrid / hybrid_rerank` 检索链路，但我不会把它说成完整生产级 RAG 平台。核心设计是先守住 `/api/search` 边界、来源时间字段和 `SearchResponse` 契约，让前端只消费统一结果结构；当前默认策略已经收口为 `hybrid`，后续重点转向监控、调度和后台治理。

## 一句话总结

如果只用一句话总结这个项目，我会说：

> 这是一个把 RAG 主题做成前端产品原型的项目，我重点解决的不是“模型会不会回答”，而是“回答如何更可信、更可核验、更像真实产品”。
