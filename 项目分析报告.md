# 校园信息检索与可解释问答助手 - 项目分析报告

## 1. 项目概述

### 1.1 项目定位

这是一个**基于 RAG（检索增强生成）思路的校园信息检索与可解释问答助手前端产品原型**。项目重点不在于构建一个聊天机器人，而在于通过优秀的前端交互设计，将问答结果、检索证据和来源可信度清晰地组织呈现给用户。

### 1.2 核心目标

- 解决校园信息查询场景中官方规则与社区经验混杂的问题
- 提供可追溯、可核验的问答体验
- 在信息不足时不强行生成答案，确保产品可信度

### 1.3 适用场景

- 学生快速查询校园规则、办事流程和生活信息
- 作为前端开发者的作品集项目，展示可解释 AI 产品的交互设计能力
- 面试中讲解状态设计、可信度表达和接口边界的案例

## 4. 数据契约设计

### 4.1 核心类型定义

#### 4.1.1 SearchSource（搜索来源）

```typescript
type SearchSource = {
  id: string;
  title: string;
  type: "official" | "community";  // 来源类型分层
  sourceName: string;
  sourceDomain?: string;
  publishedAt: string | null;      // 原始发布时间
  updatedAt: string | null;        // 来源页面更新时间
  fetchedAt: string;               // 抓取时间
  lastVerifiedAt: string | null;   // 最近校验时间
  snippet: string;
  fullSnippet?: string;
  matchedKeywords: string[];
  url?: string;
  canonicalUrl?: string;
  freshnessLabel: "fresh" | "recent" | "stale" | "undated";
  trustScore?: number;
  dedupKey?: string;
};
```

#### 4.1.2 SearchAnswer（搜索回答）

```typescript
type SearchAnswer = {
  summary: string;
  sourceNote: string;
  disclaimer: string;
  confidence: number;
  evidence?: SearchAnswerEvidence[];  // 引用证据
};
```

#### 4.1.3 SearchResponse（统一搜索响应）

```typescript
type SearchResponse = {
  query: string;
  status: "ok" | "partial" | "empty" | "error";  // 多状态设计
  answer: SearchAnswer | null;
  sources: SearchSource[];
  relatedQuestions: string[];
  retrievedCount: number;
  resultGeneratedAt: string;  // 本次响应生成时间
};
```

### 4.2 时间字段语义设计

项目对时间字段做了明确的语义区分，避免混淆：

| 字段 | 含义 |
|------|------|
| `publishedAt` | 来源原始发布时间 |
| `updatedAt` | 来源页面最近更新时间 |
| `fetchedAt` | 搜索服务抓取或入库时间 |
| `lastVerifiedAt` | 最近一次校验时间 |
| `resultGeneratedAt` | 本次回答对象生成时间 |

## 5. 检索链路

### 5.1 完整检索流程

```
用户提问
    ↓
useSearchNavigation 导航到 /search?q=...
    ↓
ResultsShell 组件发起请求
    ↓
/src/app/api/search/route.ts
    ↓
searchServiceProvider 调用外部搜索服务
    ↓
search-service 处理请求
    ├─→ 有 DATABASE_URL → Postgres chunk 检索
    └─→ 无数据库 → Seed corpus 降级
    ↓
可选：LLM 生成回答（配置 SEARCH_ANSWER_MODE=llm）
    ↓
SearchResponse 返回
    ↓
AnswerPanel / SourceList 渲染
```

### 5.2 搜索服务提供者实现

[`searchServiceProvider`](file:///workspace/src/lib/search/search-provider.ts) 负责：

- 构建搜索服务请求（支持 GET/POST）
- 处理请求超时（默认 8 秒）
- 错误分类与日志记录
- 响应归一化
- 认证头处理

## 6. 数据管道

### 6.1 来源注册表

位于 [`src/lib/search/source-registry.ts`](file:///workspace/src/lib/search/source-registry.ts)，定义：

- 官方/社区来源白名单
- 抓取方式（api/rss/html/sitemap/manual）
- 更新频率（hourly/daily/weekly/manual）
- 清洗规则配置
- 信任权重

### 6.2 清洗规则

官方来源清洗：
- 去除导航、页脚、广告
- 保留标题、发布时间、更新时间
- 合并重复段落

社区来源清洗：
- 去除广告、联系方式
- 只保留公开帖子正文
- 隐藏敏感信息（手机号、邮箱等）

### 6.3 去重策略

按优先级顺序：
1. `canonical_url` - 同一规范链接只保留一份
2. `title_and_date` - 标题和发布日期一致时标记重复
3. `content_hash` - 清洗后正文哈希一致时合并版本
4. `simhash` - 正文高相似片段在分块阶段合并

### 6.4 入库流程

```
定时任务
    ↓
来源抓取（fetch）
    ↓
正文抽取与清洗（clean）
    ↓
去重处理（dedup）
    ↓
文本分块（chunk）
    ↓
索引写入（index）
    ├─→ 全文索引
    └─→ 可选：pgvector 向量索引
    ↓
查询 API 暴露（publish）
```

## 7. 数据库设计

### 7.1 核心表结构

最小可用表集合：

- `source_registry` - 来源注册表
- `documents` - 文档主表
- `document_versions` - 文档版本
- `chunks` - 文本分块
- `ingestion_runs` - 摄取运行记录

### 7.2 pgvector 扩展

可选的向量检索增强：

```sql
-- 创建扩展
CREATE EXTENSION IF NOT EXISTS vector;

-- 给 chunks 增加向量字段
ALTER TABLE chunks ADD COLUMN embedding vector(1536);
ALTER TABLE chunks ADD COLUMN embedding_model TEXT;
ALTER TABLE chunks ADD COLUMN embedded_at TIMESTAMPTZ;

-- 创建索引（HNSW 优先，失败回退 IVFFLAT）
CREATE INDEX ON chunks USING hnsw (embedding vector_cosine_ops);
```

## 10. 项目亮点

### 10.1 可解释性设计

- **问答/检索双模式** - 用户既可以看摘要，也可以审查原始证据
- **引用可追溯** - 每条回答都对应具体的来源片段
- **时间透明度** - 显式展示发布、更新、抓取、校验时间

### 10.2 可信度控制

- **官方/社区来源分层** - 事实类优先官方，社区作为经验补充
- **多状态结果** - ok/partial/empty/error 四种状态，不强行生成答案
- **信任权重** - 不同来源有不同的 trustScore

### 10.3 工程边界清晰

- **统一 API 入口** - `/api/search` 作为前端唯一入口
- **契约先行** - 先定义 SearchResponse，再实现具体逻辑
- **降级策略** - 无数据库时使用 seed corpus，无 LLM key 时使用 extractive answer

### 10.4 状态设计完整

- 两阶段加载（retrieving → summarizing）
- 来源筛选
- 视图切换
- 来源展开
- 本地历史
- 主题切换

## 13. 面试参考要点

### 13.1 核心讲法

> 这是一个基于 RAG 思路的校园信息检索前端产品原型。我重点解决的不是"模型会不会回答"，而是"回答如何更可信、更可核验、更像真实产品"。

### 13.2 高频问题准备

| 问题 | 回答要点 |
|------|----------|
| 为什么不做聊天壳子？ | 校园信息查询更需要"可核验"而非"能生成文本" |
| 为什么分 official/community？ | 两类信息可信度和使用方式不同，避免误导 |
| 为什么有 ok/partial/empty/error？ | 真实产品不能把"没找到"和"找到一点点"混成一种状态 |
| 当前算不算真实 RAG？ | 不算生产级，是 RAG 思路驱动的前端原型 |
| 难点是什么？ | 状态完整性、AI 产品中前端价值、原型到产品的迁移边界 |

## 14. 总结

这是一个设计优秀、架构清晰的前端产品原型项目。它通过：

1. **清晰的分层架构** - 视图层、状态层、数据适配层职责明确
2. **完整的数据契约** - SearchResponse 定义了统一的接口边界
3. **可解释的交互设计** - 双视图、来源分层、引用追溯
4. **完善的测试体系** - 从单元到 E2E 的完整测试覆盖
5. **合理的降级策略** - Seed corpus、extractive answer 等降级方案

展示了前端开发者在构建 AI 产品时，如何通过优秀的交互设计和工程实践，提升产品的可信度和用户体验。
